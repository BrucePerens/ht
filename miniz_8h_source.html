15 * Low-level Deflate/Inflate implementation notes:
16
17 Compression: Use the "tdefl" API's. The compressor supports raw, static, and dynamic blocks, lazy or
18 greedy parsing, match length filtering, RLE-only, and Huffman-only streams. It performs and compresses
19 approximately as well as zlib.
20
21 Decompression: Use the "tinfl" API's. The entire decompressor is implemented as a single function
22 coroutine: see tinfl_decompress(). It supports decompression into a 32KB (or larger power of 2) wrapping buffer, or into a memory
23 block large enough to hold the entire file.
24
25 The low-level tdefl/tinfl API's do not make any use of dynamic memory allocation.
26
27 * zlib-style API notes:
28
29 miniz.c implements a fairly large subset of zlib. There's enough functionality present for it to be a drop-in
30 zlib replacement in many apps:
31 The z_stream struct, optional memory allocation callbacks
32 deflateInit/deflateInit2/deflate/deflateReset/deflateEnd/deflateBound
33 inflateInit/inflateInit2/inflate/inflateReset/inflateEnd
34 compress, compress2, compressBound, uncompress
35 CRC-32, Adler-32 - Using modern, minimal code size, CPU cache friendly routines.
36 Supports raw deflate streams or standard zlib streams with adler-32 checking.
37
38 Limitations:
39 The callback API's are not implemented yet. No support for gzip headers or zlib static dictionaries.
40 I've tried to closely emulate zlib's various flavors of stream flushing and return status codes, but
41 there are no guarantees that miniz.c pulls this off perfectly.
42
43 * PNG writing: See the tdefl_write_image_to_png_file_in_memory() function, originally written by
44 Alex Evans. Supports 1-4 bytes/pixel images.
45
46 * ZIP archive API notes:
47
48 The ZIP archive API's where designed with simplicity and efficiency in mind, with just enough abstraction to
49 get the job done with minimal fuss. There are simple API's to retrieve file information, read files from
50 existing archives, create new archives, append new files to existing archives, or clone archive data from
51 one archive to another. It supports archives located in memory or the heap, on disk (using stdio.h),
52 or you can specify custom file read/write callbacks.
53
54 - Archive reading: Just call this function to read a single file from a disk archive:
55
56 void *mz_zip_extract_archive_file_to_heap(const char *pZip_filename, const char *pArchive_name,
57 size_t *pSize, mz_uint zip_flags);
58
59 For more complex cases, use the "mz_zip_reader" functions. Upon opening an archive, the entire central
60 directory is located and read as-is into memory, and subsequent file access only occurs when reading individual files.
61
62 - Archives file scanning: The simple way is to use this function to scan a loaded archive for a specific file:
63
64 int mz_zip_reader_locate_file(mz_zip_archive *pZip, const char *pName, const char *pComment, mz_uint flags);
65
66 The locate operation can optionally check file comments too, which (as one example) can be used to identify
67 multiple versions of the same file in an archive. This function uses a simple linear search through the central
68 directory, so it's not very fast.
69
70 Alternately, you can iterate through all the files in an archive (using mz_zip_reader_get_num_files()) and
71 retrieve detailed info on each file by calling mz_zip_reader_file_stat().
72
73 - Archive creation: Use the "mz_zip_writer" functions. The ZIP writer immediately writes compressed file data
74 to disk and builds an exact image of the central directory in memory. The central directory image is written
75 all at once at the end of the archive file when the archive is finalized.
76
77 The archive writer can optionally align each file's local header and file data to any power of 2 alignment,
78 which can be useful when the archive will be read from optical media. Also, the writer supports placing
79 arbitrary data blobs at the very beginning of ZIP archives. Archives written using either feature are still
80 readable by any ZIP tool.
81
82 - Archive appending: The simple way to add a single file to an archive is to call this function:
83
84 mz_bool mz_zip_add_mem_to_archive_file_in_place(const char *pZip_filename, const char *pArchive_name,
85 const void *pBuf, size_t buf_size, const void *pComment, mz_uint16 comment_size, mz_uint level_and_flags);
86
87 The archive will be created if it doesn't already exist, otherwise it'll be appended to.
88 Note the appending is done in-place and is not an atomic operation, so if something goes wrong
89 during the operation it's possible the archive could be left without a central directory (although the local
90 file headers and file data will be fine, so the archive will be recoverable).
91
92 For more complex archive modification scenarios:
93 1. The safest way is to use a mz_zip_reader to read the existing archive, cloning only those bits you want to
94 preserve into a new archive using using the mz_zip_writer_add_from_zip_reader() function (which compiles the
95 compressed file data as-is). When you're done, delete the old archive and rename the newly written archive, and
96 you're done. This is safe but requires a bunch of temporary disk space or heap memory.
97
98 2. Or, you can convert an mz_zip_reader in-place to an mz_zip_writer using mz_zip_writer_init_from_reader(),
99 append new files as needed, then finalize the archive which will write an updated central directory to the
100 original archive. (This is basically what mz_zip_add_mem_to_archive_file_in_place() does.) There's a
101 possibility that the archive's central directory could be lost with this method if anything goes wrong, though.
102
103 - ZIP archive support limitations:
104 No spanning support. Extraction functions can only handle unencrypted, stored or deflated files.
105 Requires streams capable of seeking.
106

118*/
